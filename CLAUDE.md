# 🔥 НА УГЛЕ — CLAUDE.md

> Pop-up гриль-бистро на набережной Самары. Летний сезон.
> Стек: Laravel + React (Inertia.js SSR) + Orchid + MySQL

---

## 🧠 МУЛЬТИАГЕНТНАЯ КОМАНДА

Этот проект управляется командой из 7 специализированных агентов.
Каждый агент имеет свою роль, зону ответственности и стиль мышления.
Перед любой задачей — определи, какой агент её выполняет, и думай как он.

---

## 👥 РОЛИ АГЕНТОВ

### 🏗️ ARCHITECT
**Роль:** Системный архитектор проекта
**Думает как:** Senior Laravel developer с опытом SSR-проектов

**Зона ответственности:**
- Структура базы данных (MySQL + Eloquent модели)
- Архитектура маршрутов Laravel + Inertia.js
- Организация папок и неймспейсов
- API-контракты между бэкендом и React-компонентами
- Производительность и кэширование

**Модели данных проекта:**
- `MenuItem` — блюдо (название, описание, цена, категория, фото, is_featured, is_available)
- `MenuCategory` — категория (название, slug, sort_order, icon)
- `GalleryPhoto` — фото галереи (url, alt, sort_order)
- `TeamMember` — член команды (имя, роль, bio, фото)
- `SiteSettings` — настройки (часы работы, адрес, сезон, контакты)
- `Reservation` — заявка на бронь (имя, телефон, дата, кол-во гостей, комментарий)

**Правила:**
- Все модели — Eloquent, никаких raw SQL без крайней необходимости
- SSR через `@inertiajs/react` — компоненты должны рендериться на сервере
- Env-переменные для всех секретов, никаких хардкодов

### ⚙️ BACKEND
**Роль:** Laravel-разработчик
**Думает как:** Чистый код, SOLID, минимум зависимостей

**Зона ответственности:**
- Controllers (тонкие, логика в Services)
- Eloquent модели и Relations
- Form Requests для валидации
- Inertia responses (`Inertia::render(...)`)
- Очереди для отправки писем/уведомлений

**Маршруты проекта:**
```php
// Публичные
GET  /                  → HomeController@index      (лендинг)
GET  /menu              → MenuController@index      (полное меню)
POST /reservation       → ReservationController@store

// Orchid Admin
/admin/*                → Orchid Dashboard
```

**Правила:**
- Все данные для Inertia передавать через `props` — никакого прямого blade в React-зонах
- Валидация — только через FormRequest классы
- Ответы на ошибки — JSON для API, Inertia redirect для форм

### ⚛️ FRONTEND
**Роль:** React-разработчик (Inertia.js)
**Думает как:** Компонентный мыслитель, UX-ориентированный

**Зона ответственности:**
- React-компоненты страниц (Pages/)
- Переиспользуемые компоненты (Components/)
- Анимации и микровзаимодействия
- SSR-совместимость (никакого window/document без проверки)

**Структура компонентов:**
```
resources/js/
├── Pages/
│   ├── Home.jsx          ← главный лендинг
│   └── Menu.jsx          ← страница меню
├── Components/
│   ├── Hero/
│   ├── MenuSection/
│   ├── ChefSection/
│   ├── GallerySection/
│   ├── ReservationForm/
│   ├── LocationSection/
│   └── Layout/
└── app.jsx
```

**Секции лендинга (порядок):**
1. `<Hero>` — корова + название + tagline + CTA
2. `<AboutSection>` — о проекте (3-4 предложения)
3. `<ChefSection>` — Андрей Воробьёв + история + La Volte коллаба
4. `<MenuSection>` — хиты по категориям (не всё меню)
5. `<GallerySection>` — живые фото с проекта
6. `<LocationSection>` — адрес, карта, сезон, часы
7. `<ReservationForm>` — форма для групп
8. `<Footer>` — минимум: контакты, соцсети

**Правила:**
- Tailwind для стилей + CSS-переменные для брендовых цветов
- Lazy loading для изображений галереи
- Form state через React useState, submit через Inertia.useForm
- Анимации появления — Intersection Observer + CSS transitions

### 📋 ADMIN
**Роль:** Orchid-разработчик
**Думает как:** Удобство для владельца ресторана, не для разработчика

**Зона ответственности:**
- Orchid Screens и Resources
- CRUD для всего контента
- Dashboard с ключевыми метриками

**Orchid-экраны:**
```
MenuItemResource    → управление блюдами (вкл/выкл, редактирование)
MenuCategoryResource → категории и их порядок
GalleryScreen       → загрузка и сортировка фото
TeamScreen          → команда и шеф
ReservationScreen   → просмотр заявок на бронь
SiteSettingsScreen  → часы работы, адрес, сезон (открыт/закрыт)
```

**Главная фишка:** кнопка "Ресторан открыт/закрыт" прямо на дашборде.
Один клик — и на сайте появляется/исчезает баннер "Сегодня мы работаем".

**Правила:**
- Поля Orchid должны быть понятны нетехническому пользователю
- Изображения — через Orchid Attachments + Storage
- Сортировка через drag-and-drop где возможно

### 🚀 DEVOPS
**Роль:** Инженер деплоя и инфраструктуры
**Думает как:** Надёжность и простота обслуживания

**Зона ответственности:**
- Настройка VDS (Ubuntu + Nginx + PHP-FPM + Node.js)
- SSL через Let's Encrypt
- Деплой через GitHub Actions или ручной git pull
- .env для продакшна
- MySQL бекапы
- Supervisor для очередей Laravel

**Стек сервера:**
```
OS:       Ubuntu 22.04 LTS
Web:      Nginx
PHP:      8.2+ (php-fpm)
Node:     18+ (для SSR Inertia)
DB:       MySQL 8.0
Queue:    Redis + Laravel Horizon (или database queue для старта)
SSL:      Certbot / Let's Encrypt
```

**Команды деплоя:**
```bash
composer install --no-dev --optimize-autoloader
php artisan config:cache
php artisan route:cache
php artisan view:cache
php artisan migrate --force
npm ci && npm run build
sudo supervisorctl restart all
```

**Правила:**
- Никаких секретов в репозитории
- .env.example всегда актуален
- Логи ошибок — в storage/logs, ротация настроена
- Инструкция по деплою в README.md

## 🚫 СТОП-ПРАВИЛА

1. **Никаких PDF-меню** — только живые React-компоненты с данными из БД
2. **Никаких стоковых фото** — только реальные фото проекта (placeholder → реальное)
3. **Никакого fine dining tone** — сайт демократичный, живой, с характером
4. **Никаких лишних страниц** — это лендинг, не порtal
5. **Mobile-first всегда** — сначала телефон, потом десктоп
6. **Orchid для всего контента** — ни один текст не должен быть захардкожен в коде
7. **SSR обязателен** — Inertia SSR настроен, не отключать
